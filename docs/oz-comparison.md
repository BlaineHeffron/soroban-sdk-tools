# soroban-sdk-tools vs OpenZeppelin Stellar Contracts: A Side-by-Side Comparison

This document compares the composability approaches of OpenZeppelin's `stellar-contracts` (v0.6.0) with `soroban-sdk-tools`' `#[contracttrait]` macro. The goal is to demonstrate how the same patterns can be achieved with stronger structural guarantees and better developer ergonomics.

---

## Ownable: Side-by-Side

### OpenZeppelin Approach (~80 lines across 3 files)

```rust
// === packages/access/src/ownable/storage.rs ===
#[contracttype]
pub enum OwnableStorageKey { Owner, PendingOwner }

pub fn get_owner(e: &Env) -> Option<Address> {
    e.storage().instance().get::<_, Address>(&OwnableStorageKey::Owner)
}

pub fn set_owner(e: &Env, owner: &Address) {
    if e.storage().instance().has(&OwnableStorageKey::Owner) {
        panic_with_error!(e, OwnableError::OwnerAlreadySet);
    }
    e.storage().instance().set(&OwnableStorageKey::Owner, &owner);
}

pub fn transfer_ownership(e: &Env, new_owner: &Address, live_until_ledger: u32) {
    let owner = enforce_owner_auth(e);
    transfer_role(e, new_owner, &OwnableStorageKey::PendingOwner, live_until_ledger);
    emit_ownership_transfer(e, &owner, new_owner, live_until_ledger);
}

pub fn accept_ownership(e: &Env) {
    let new_owner = accept_transfer(e, &OwnableStorageKey::Owner, &OwnableStorageKey::PendingOwner);
    emit_ownership_transfer_completed(e, &new_owner);
}

pub fn enforce_owner_auth(e: &Env) -> Address {
    let Some(owner) = get_owner(e) else {
        panic_with_error!(e, OwnableError::OwnerNotSet);
    };
    owner.require_auth();
    owner
}

// === packages/access/src/ownable/mod.rs ===
#[contracttrait]
pub trait Ownable {
    fn get_owner(e: &Env) -> Option<Address> { get_owner(e) }
    fn transfer_ownership(e: &Env, new_owner: Address, live_until_ledger: u32) {
        transfer_ownership(e, &new_owner, live_until_ledger);
    }
    fn accept_ownership(e: &Env) { accept_ownership(e); }
    fn renounce_ownership(e: &Env) { renounce_ownership(e); }
}

// === Contract usage ===
#[contractimpl(contracttrait)]
impl Ownable for MyContract {}

// === Auth check in other methods (manual) ===
#[contractimpl]
impl MyContract {
    #[only_owner]  // OZ macro -- injects enforce_owner_auth(e)
    pub fn admin_action(e: &Env) { /* ... */ }
}
```

**Characteristics:**
- Auth is in module-level functions (`enforce_owner_auth`), not in the trait
- `#[only_owner]` macro is a separate concern, applied per-method
- Two-step transfer (transfer + accept) -- excellent safety pattern
- No DI / provider swapping -- single implementation baked into module functions
- Storage keys are explicit `#[contracttype]` enums

### soroban-sdk-tools Approach (~35 lines in 1 file)

```rust
// === Trait definition (all-in-one) ===
#[contracttrait]
pub trait Ownable {
    fn owner(env: &Env) -> Address;

    #[auth(Self::owner)]
    fn transfer_ownership(env: &Env, new_owner: Address);
}

// === Generated by macro ===
// - OwnableInternal trait (pure business logic)
// - Ownable trait (auth-wrapped, type Provider for DI)
// - OwnableAuthClient (test helper)
// - impl_ownable! macro (sealed auth)

// === Provider (swappable implementation) ===
pub struct SingleOwner;
impl OwnableInternal for SingleOwner {
    fn owner(env: &Env) -> Address {
        OwnableStorage::get_owner(env).expect("not initialized")
    }
    fn transfer_ownership(env: &Env, new_owner: Address) {
        OwnableStorage::set_owner(env, &new_owner);
    }
}

// === Contract wiring ===
// Option A: Sealed (auth cannot be overridden)
impl_ownable!(MyContract, SingleOwner);

// Option B: Flexible (auth can be customized)
#[contractimpl(contracttrait)]
impl Ownable for MyContract {
    type Provider = SingleOwner;
}
```

**Characteristics:**
- Auth is structural -- `#[auth(Self::owner)]` generates `require_auth()` in the trait wrapper
- Provider pattern enables DI -- swap `SingleOwner` for `MultisigOwner` with one line change
- `impl_ownable!` prevents auth bypass (sealed pattern)
- Single trait definition generates everything
- Less code, stronger guarantees

### Key Differences

| Aspect | OpenZeppelin | soroban-sdk-tools |
|--------|-------------|-------------------|
| Auth enforcement | Manual per-method (`#[only_owner]` macro) | Structural in trait definition (`#[auth]`) |
| Implementation swapping | Not supported | `type Provider = X` (one-line swap) |
| Override protection | None (defaults can be overridden) | `impl_ownable!` macro (sealed) |
| Two-step transfer | Built-in | Can be added in provider |
| Lines of code (library) | ~80 across 3 files | ~35 in 1 file + macro |
| Lines of code (consumer) | ~5 | ~5 |
| Event emission | Built-in | Convention (provider responsibility) |
| Testing | `mock_all_auths()` | `OwnableAuthClient` with `.authorize().invoke()` |

## Pausable: Side-by-Side

### OpenZeppelin

```rust
// Trait -- deliberately has NO default for pause/unpause (auth is left to consumer)
pub trait Pausable {
    fn paused(e: &Env) -> bool { pausable::paused(e) }
    fn pause(e: &Env, caller: Address);   // NO default -- consumer must implement
    fn unpause(e: &Env, caller: Address); // NO default -- consumer must implement
}

// Consumer must wire auth manually:
#[contractimpl]
impl Pausable for MyContract {
    fn pause(e: &Env, caller: Address) {
        caller.require_auth();
        let owner: Address = e.storage().instance().get(&OWNER).expect("...");
        if owner != caller { panic_with_error!(e, Error::Unauthorized); }
        pausable::pause(e);
    }
    fn unpause(e: &Env, caller: Address) {
        caller.require_auth();
        let owner: Address = e.storage().instance().get(&OWNER).expect("...");
        if owner != caller { panic_with_error!(e, Error::Unauthorized); }
        pausable::unpause(e);
    }
}
```

### soroban-sdk-tools

```rust
// Trait -- auth is structural via supertrait
#[contracttrait]
pub trait Pausable: Ownable {
    fn is_paused(env: &Env) -> bool;

    #[auth(Self::owner)]  // auth comes from Ownable's owner()
    fn pause(env: &Env);

    #[auth(Self::owner)]
    fn unpause(env: &Env);
}

// Consumer -- zero auth boilerplate:
#[contractimpl(contracttrait)]
impl Pausable for MyContract {
    type Provider = SingleOwner;  // same provider handles both traits
}
```

**The difference is stark:** OZ requires 12 lines of manual auth checking per consumer. Our approach requires 0 -- auth is declared once in the trait and enforced everywhere.

## FungibleToken + ContractOverrides: Side-by-Side

### OpenZeppelin's ContractOverrides Pattern

```rust
// The override trait (extension point)
pub trait ContractOverrides {
    fn transfer(e: &Env, from: &Address, to: &MuxedAddress, amount: i128) {
        Base::transfer(e, from, to, amount);
    }
    // ... 10+ methods with defaults delegating to Base
}

// The public trait
#[contracttrait]
pub trait FungibleToken {
    type ContractType: ContractOverrides;
    fn transfer(e: &Env, from: Address, to: MuxedAddress, amount: i128) {
        Self::ContractType::transfer(e, &from, &to, amount);
    }
}

// Consumer adds pause check via macro:
#[contractimpl]
impl FungibleToken for MyToken {
    type ContractType = Base;

    #[when_not_paused]  // OZ macro
    fn transfer(e: &Env, from: Address, to: MuxedAddress, amount: i128) {
        Self::ContractType::transfer(e, &from, &to, amount);
    }
}
```

### soroban-sdk-tools Equivalent

```rust
#[contracttrait]
pub trait FungibleToken {
    #[auth(from)]  // from.require_auth() is structural
    fn transfer(env: &Env, from: Address, to: Address, amount: i128);

    fn balance(env: &Env, addr: Address) -> i128;
    fn total_supply(env: &Env) -> i128;
}

// Provider with pause integration:
pub struct PausableToken;
impl FungibleTokenInternal for PausableToken {
    fn transfer(env: &Env, from: Address, to: Address, amount: i128) {
        // Pause check in provider (composable)
        assert!(!PausableStorage::get_paused(env).unwrap_or(false), "paused");
        BalanceStorage::update_balances(env, &from, |b| b.unwrap_or(0) - amount);
        BalanceStorage::update_balances(env, &to, |b| b.unwrap_or(0) + amount);
    }
    // ...
}

// Consumer:
impl_fungible_token!(MyToken, PausableToken);
```

### Key Insight

OZ uses TWO composition mechanisms that don't interact:
1. `ContractOverrides` (type-level strategy selection via `type ContractType`)
2. `#[when_not_paused]` (macro-level function decoration)

Our approach unifies both into ONE mechanism:
1. `type Provider` selects the implementation strategy
2. `#[auth]` in the trait definition handles auth structurally
3. The provider handles business concerns (pause checks, etc.)

This means a single `PausableToken` provider can integrate pause checks, allowlists, blocklists, and custom logic -- all in one place, all type-checked at compile time.

---

## What OZ Does Better (and What We Should Adopt)

1. **Two-step transfers** -- OZ's `transfer_ownership` + `accept_ownership` pattern prevents mistakes. Our providers should support this.

2. **Event emission** -- OZ emits events for every state change. We should generate events in the outer trait defaults.

3. **TTL management** -- OZ has explicit TTL constants and extension patterns. Our `#[contractstorage]` could integrate this.

4. **Error code ranges** -- OZ namespaces errors by module (Ownable: 2100+, Pausable: 1000+). Our `#[scerr]` handles this via const-chaining.

5. **Role-based access control** -- OZ's RBAC with role enumeration, role admins, and two-step admin transfer is comprehensive. This should be a provider in our system.

---

## What We Do Better

1. **Structural auth enforcement** -- OZ relies on `#[only_owner]` macros applied per-method. We enforce auth in the trait definition itself.

2. **Provider swapping** -- OZ has no DI mechanism for Ownable/Pausable. Switching from single-owner to multisig requires rewriting the module. We swap one type parameter.

3. **Sealed auth** -- OZ's trait defaults can be overridden, bypassing auth. Our `impl_ownable!` prevents this.

4. **AuthClient testing** -- OZ uses `mock_all_auths()`. We provide per-trait `AuthClient` with `.authorize().invoke()` for precise auth testing.

5. **Composable errors** -- OZ uses manual error code ranges. Our `#[scerr]` auto-chains codes across composed traits.

6. **Single definition** -- OZ requires 3 files per feature (storage.rs, mod.rs, consumer). We require 1 trait definition.
