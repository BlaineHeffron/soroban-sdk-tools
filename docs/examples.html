Side-by-side

What gets shorter, and what stays explicit.

These comparisons come from patterns already in the repository: typed storage from the increment examples, composed errors from the error tests, and auth builder flows from the token and auth test suites.

Storage

From repeated key plumbing to typed accessors

Manual pattern

#[derive(Clone)]
#[contracttype]
enum DataKey {
    Value,
    Balances(Address),
}

fn get_value(env: &Env) -> u32 {
    env.storage().instance().get(&DataKey::Value).unwrap_or(0)
}

fn set_value(env: &Env, value: u32) {
    env.storage().instance().set(&DataKey::Value, &value);
}

fn balance(env: &Env, addr: &Address) -> u32 {
    env.storage()
        .persistent()
        .get(&DataKey::Balances(addr.clone()))
        .unwrap_or(0)
}

With #[contractstorage]

#[contractstorage]
pub struct Counter {
    value: InstanceItem<u32>,
    values: PersistentMap<Address, u32>,
}

pub fn increment_by(env: &Env, amount: u32) -> u32 {
    Counter::update_value(env, |current| {
        current.unwrap_or(0).saturating_add(amount)
    })
}

pub fn increment_for(env: &Env, addr: &Address) -> u32 {
    let count = Counter::get_values(env, addr).unwrap_or(0) + 1;
    Counter::set_values(env, addr, &count);
    count
}

The contract still decides how state changes. The macro removes the repeated key definitions and read/write wrappers.

Errors

From manual numbering to generated composition

Manual pattern

#[contracterror]
#[derive(Copy, Clone, Eq, PartialEq)]
pub enum AppError {
    Unauthorized = 1,
    MathDivisionByZero = 2,
    MathNegativeInput = 3,
    ExternalCalcOverflow = 4,
}

impl From<MathError> for AppError {
    fn from(err: MathError) -> Self {
        match err {
            MathError::DivisionByZero => Self::MathDivisionByZero,
            MathError::NegativeInput => Self::MathNegativeInput,
        }
    }
}

With #[scerr]

#[scerr]
pub enum OuterError {
    /// unauthorized
    Unauthorized,

    #[transparent]
    Math(#[from] MathError),

    #[from_contract_client]
    MathClient(MathError),

    #[transparent]
    Calc(#[from] CalcError),
}

Variant order defines the sequential codes. Numbering stays predictable while a large chunk of hand-maintained conversion glue disappears.

Auth testing

From nested mock structures to builder-style calls

Manual MockAuth

env.mock_auths(&[
    soroban_sdk::testutils::MockAuth {
        address: &alice,
        invoke: &soroban_sdk::testutils::MockAuthInvoke {
            contract: &contract_id,
            fn_name: "transfer",
            args: (&alice, &bob, 300_i128).into_val(env),
            sub_invokes: &[],
        },
    },
]);

client.transfer(&alice, &bob, &300);

Generated AuthClient

token::AuthClient::new(env, &contract_id)
    .transfer(&alice, &bob, &300)
    .authorize(&alice)
    .invoke();

This matters most when two or three parties need to authorize the same call. The test reads like the action it is testing, not like the auth data structures underneath.

Reduction strategy

Code gets smaller where it should.

This is not about compressing protocol logic into magic. It strips out repetition around state access, error mapping, and authorization scaffolding. What is left is the code you actually want to read.

Net effect

  • Fewer places to make copy-paste mistakes
  • More readable examples for developers new to Soroban
  • Cleaner tests as contract auth rules grow more complex
