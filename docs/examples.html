Side-by-side

What gets shorter, and what stays explicit.

Each pair below does the same job. One just takes more scaffolding.

Storage

Same counter update, less storage plumbing

Manual pattern

#[derive(Clone)]
#[contracttype]
enum DataKey {
    Value,
}

pub fn increment_by(env: &Env, amount: u32) -> u32 {
    let current = env
        .storage()
        .instance()
        .get(&DataKey::Value)
        .unwrap_or(0);

    let next = current.saturating_add(amount);
    env.storage().instance().set(&DataKey::Value, &next);
    next
}

With #[contractstorage]

#[contractstorage]
pub struct Counter {
    value: InstanceItem<u32>,
}

pub fn increment_by(env: &Env, amount: u32) -> u32 {
    Counter::update_value(env, |current| {
        current.unwrap_or(0).saturating_add(amount)
    })
}

Both versions increment the same counter and return the result. The generated version just takes less setup.

Errors

Same cross-contract call, less error glue

Manual pattern

#[contracterror]
#[derive(Copy, Clone, Eq, PartialEq)]
pub enum AppError {
    ExternalDivisionByZero = 1,
    ExternalNegativeInput = 2,
    ExternalInvoke = 3,
}

impl From<external::MathError> for AppError {
    fn from(err: external::MathError) -> Self {
        match err {
            external::MathError::DivisionByZero => Self::ExternalDivisionByZero,
            external::MathError::NegativeInput => Self::ExternalNegativeInput,
        }
    }
}

impl From<soroban_sdk::InvokeError> for AppError {
    fn from(_: soroban_sdk::InvokeError) -> Self {
        Self::ExternalInvoke
    }
}

pub fn safe_div_with(
    env: &Env,
    address: &Address,
    num: &i64,
    denom: &i64,
) -> Result<i64, AppError> {
    match external::Client::new(env, address).try_safe_div(num, denom) {
        Ok(Ok(value)) => Ok(value),
        Ok(Err(err)) => Err(AppError::from(err)),
        Err(Ok(err)) => Err(AppError::from(err)),
        Err(Err(invoke_err)) => Err(AppError::from(invoke_err)),
    }
}

With #[scerr]

#[scerr]
pub enum Error {
    #[from_contract_client]
    Math(#[from] external::MathError),
}

pub fn safe_div_with(
    env: &Env,
    address: &Address,
    num: &i64,
    denom: &i64,
) -> Result<i64, Error> {
    let result = external::Client::new(env, address)
        .try_safe_div(num, denom);
    Ok(result??)
}

Both paths call the same external contract. #[scerr] lets you keep the try_* ?? path; the manual version has to unpack both layers by hand.

Numbered ranges

Same grouped native codes, less manual bookkeeping

Anchored basic enum

#[scerr]
pub enum ContractErrors {
    UnexpectedError = 1,

    UnauthorizedSigner = 100,
    WrongVoter,
    MaintainerNotDomainOwner,

    InvalidKey = 200,
    ProjectAlreadyExist,

    AlreadyVoted = 400,
    ProposalVotingTime,
}

Wrapped later

#[scerr]
pub enum AppError {
    Unauthorized,

    #[transparent]
    Contract(#[from] ContractErrors),
}

// External/native codes stay 1, 100, 101, 200, 400...
// Inner composition still uses a dense sequence internally.

Use explicit discriminants when downstream consumers care about stable grouped ranges. #[scerr] still keeps nested composition dense, so wrappers and imported errors continue to work.

Auth testing

Same token transfer, less auth setup

Manual MockAuth

env.mock_auths(&[
    soroban_sdk::testutils::MockAuth {
        address: &alice,
        invoke: &soroban_sdk::testutils::MockAuthInvoke {
            contract: &contract_id,
            fn_name: "transfer",
            args: (&alice, &bob, 300_i128).into_val(env),
            sub_invokes: &[],
        },
    },
]);

client.transfer(&alice, &bob, &300);

Generated AuthClient

#[test]
fn transfer_auth_flow() {
    let env = Env::default();
    token::AuthClient::new(&env, &contract_id)
        .transfer(&alice, &bob, &300)
        .authorize(&alice)
        .invoke();
}

Both tests authorize the same transfer. The builder version reads more like the call you're trying to test.

Nested auth

Same nested invocation, less auth tree boilerplate

Manual nested MockAuthInvoke

#[test]
fn trade_with_nested_auth_manual() {
    let transfer_sub = soroban_sdk::testutils::MockAuthInvoke {
        contract: &token_id,
        fn_name: "transfer_from",
        args: (&buyer, &buyer, &seller, &amount).into_val(&env),
        sub_invokes: &[],
    };

    let trade_root = soroban_sdk::testutils::MockAuthInvoke {
        contract: &market_id,
        fn_name: "trade",
        args: (&buyer, &seller, &offer_id, &amount).into_val(&env),
        sub_invokes: &[transfer_sub],
    };

    env.mock_auths(&[
        soroban_sdk::testutils::MockAuth { address: &buyer, invoke: &trade_root },
        soroban_sdk::testutils::MockAuth { address: &seller, invoke: &trade_root },
    ]);

    market::Client::new(&env, &market_id)
        .trade(&buyer, &seller, &offer_id, &amount);
}

With AuthClient

#[test]
fn trade_with_nested_auth() {
    market::AuthClient::new(&env, &market_id)
        .trade(&buyer, &seller, &offer_id, &amount)
        .authorize(&buyer)
        .authorize(&seller)
        .invoke();
}

Both tests authorize the same trade call. The manual version builds the nested invoke tree directly; AuthClient keeps the test focused on who authorizes the call.

Reduction strategy

Code gets smaller where it should.

The point is simple: less repetition around storage, errors, and auth.

Net effect

  • Fewer copy-paste mistakes
  • More readable examples
  • Cleaner auth tests
