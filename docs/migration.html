Incremental adoption

Migrate one concern at a time.

No rewrite needed. The cleanest path is usually storage first, then errors, then auth tests. Each step removes repeated code without forcing a new contract architecture.

Recommended sequence

  1. Adopt #[contractstorage] for the state that changes most often.
  2. Replace hand-numbered enums with #[scerr] once error boundaries are clear.
  3. Switch tests from manual MockAuth trees to AuthClient builders.
  4. Use real signer helpers only where signature behavior is part of what you are testing.

Step 1

Storage migration

What to replace

Manual key enums, repeated env.storage() calls, one-off helper functions, and duplicated get/set/update logic scattered across contract methods.

What to introduce

A storage struct with explicit item and map types, annotated with #[contractstorage]. Use auto_shorten = true if you want generated short keys by default.

Checklist

  • Move your storage keys into a single struct definition.
  • Choose durability explicitly: persistent, instance, or temporary.
  • Swap helper functions for generated methods like get_field, set_field, and update_field.
  • Keep Storage::new(env) when you want grouped access inside one method.

Watch for

  • Key names become part of your storage layout. Treat them like schema.
  • Changing durability or key shape is still a state migration, not a macro toggle.
  • auto_shorten is helpful, but be deliberate when existing contracts already rely on stable keys.

Step 2

Error migration

What to replace

Enums with manually assigned numeric codes, custom conversion glue, and error composition logic that gets harder to maintain as contract boundaries grow.

What to introduce

#[scerr] on your contract error enums, then transparent or contract-client variants for composed errors once you have imported client types.

Checklist

  • Convert one enum at a time and drop the manual numeric assignments.
  • Move human-readable descriptions into doc comments on each variant.
  • For same-WASM composition, use #[transparent] with #[from].
  • For client try_ calls, use #[from_contract_client] and propagate with ??.

Watch for

  • Sequential codes are generated in enum order — reordering variants is a breaking change.
  • Cross-contract composition only works cleanly when imported errors implement the generated traits.
  • Keep old numeric mappings documented when migrating a live protocol surface.

Step 3

Auth test migration

What to replace

Repeated MockAuth arrays with copied contract IDs, function names, serialized args, and manually nested invoke trees spread across multiple tests.

What to introduce

The generated AuthClient from contractimport!, plus setup_mock_auth and setup_real_auth when you want helpers outside the builder pattern.

Checklist

  • Import contract WASM with contractimport! in tests.
  • Create an AuthClient alongside the normal read/query client.
  • Replace manual auth trees with chained .authorize() calls for mock auth.
  • Use .sign() only for cases where cryptographic signer behavior matters.

Watch for

  • Mock auth is still the right default for setup and non-auth-focused tests.
  • Multi-signer contracts are where the readability gain is largest.
  • Real signer tests cost more setup. Save them for signature semantics, not every happy path.

Migration principle

Use the tools to remove repetition, not to hide domain logic.

Your storage schema, protocol invariants, and authorization rules should stay visible in contract code. The only thing that goes away is the boilerplate around them.
