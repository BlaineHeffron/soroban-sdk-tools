Soroban developer tooling

Build contracts, not scaffolding.

soroban-sdk-tools handles the repetitive parts of Soroban development: storage wiring, contract error definitions, and auth test setup. You keep writing normal Soroban code, but the boilerplate around it shrinks.

Migration guide See the code

Install

cargo add soroban-sdk-tools
cargo add soroban-sdk-tools --dev --features testutils
What it replaces Manual storage keys, hand-numbered errors, verbose auth mocks
What you keep Standard Soroban contract code, explicit state transitions, your existing tests
Why it matters Faster onboarding for Rust developers new to Soroban

Storage

Typed accessors instead of hand-rolled key enums

Drop #[contractstorage] on a struct and get generated storage handles, accessors, and update helpers. Works across persistent, instance, and temporary storage. Turn on auto_shorten to compress keys and cut storage costs by 30–40%.

  • PersistentMap, InstanceItem, and friends stay explicit
  • One-liner helpers for simple reads and updates
  • Struct-style access when you need several fields at once

Errors

Composable error enums with predictable codes

#[scerr] generates contract errors with sequential codes and handles error composition for cross-contract calls. You stop manually numbering variants and writing conversion boilerplate.

  • Doc comments become user-facing descriptions
  • #[transparent] and #[from_contract_client] for multi-contract patterns
  • Generated specs stay flattened for downstream tooling

Auth testing

AuthClient kills the worst test boilerplate

contractimport! generates an AuthClient that reads like the contract call it exercises. Chain .authorize() or .sign(), then invoke. Hand-building nested mock auth entries for every test case becomes a thing of the past.

  • Mock auth and real signature flows both supported
  • Multi-signer chains stay readable
  • Works with ed25519, secp256k1, and secp256r1

Why this project exists

The ecosystem grows when onboarding gets easier.

Experienced Rust developers should be able to pick up Soroban without first writing their own macro layer. Boilerplate slows that process down, makes examples harder to follow, and adds maintenance weight to every contract team.

This project strips out that overhead without hiding Soroban itself. The API stays close to the SDK. The repetitive parts just get generated and typed.

Good fit

  • Teams standardizing storage patterns across multiple contracts
  • Projects composing errors across client and contract boundaries
  • Test suites with frequent auth mocking or signer setup

Design principle

Remove repetition, not control. Your contract code stays explicit.
