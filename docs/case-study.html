Live measurement report

Three storage-key routes, measured on Soroban mainnet.

We deployed nine contracts across three small contract families and measured the XDR byte length of the keys they actually use on-chain.

Top line

Auto shortening wins once addresses enter the key.

Method

How the measurements were produced

  1. Built 9 separate WASM contracts from a dedicated case-study workspace in this repository.
  2. Deployed every variant to Soroban mainnet on April 13, 2026 using the local mainnet-study-headless source account.
  3. Invoked representative write paths first, then called dedicated measurement methods on the deployed contracts.
  4. Each measurement method returned key.to_xdr(env).len(), so the reported sizes are the actual XDR byte lengths for that deployed variant.

What was compared

Manual Soroban SDK storage keys, auto_shorten=true, and auto_shorten=true, symbolic=true.

Why these contracts

increment covers one-item storage, account covers stored account metadata, and token-lite covers address and address-pair map keys.

Scope

This page is about key size, not total transaction cost. Keeping that separate makes the storage comparison easier to trust.

Findings

What the data says

  • For single-item instance storage, the two tools-based routes ended up at the same size.
  • For one-address map keys, symbolic mode kept readable prefixes but still cost noticeably more than the hashed route.
  • For two-address composite keys, the fully automatic route pulled away fast.
