# Review: soroban-sdk-tools -- Post-Quantum Cryptography & Future-Proofing Assessment

**Reviewer:** Dr. Elara Vasquez
**Background:** Quantum computing researcher at ETH Zurich; focus on lattice-based cryptography and migration strategies for classical-to-post-quantum transitions; published 30+ papers on cryptographic agility
**Focus:** Future-proofing the auth model, migration paths when current cryptography breaks

---

## Executive Summary

The soroban-sdk-tools `#[contracttrait]` macro presents a composability model with a deeply embedded assumption: that `Address.require_auth()` will remain a secure primitive indefinitely. This assumption is architecturally sound today but creates a tightly coupled dependency on whatever cryptographic scheme underlies Soroban's auth system. My review examines what happens when -- not if -- Ed25519 and secp256k1 are broken by cryptographically relevant quantum computers (CRQCs), and whether soroban-sdk-tools' architecture can survive the transition.

---

## 1. The Cryptographic Surface Area

### What soroban-sdk-tools depends on

The entire auth model rests on a single primitive:

```rust
__auth_addr.require_auth();
```

This call, generated by the `#[auth(Self::owner)]` macro, delegates to Soroban's runtime for signature verification. The framework itself performs zero cryptographic operations. This is both a strength and a concern.

**Strength:** The framework is cryptographically agnostic. It does not implement, select, or constrain the signature scheme. If Soroban migrates to post-quantum signatures, soroban-sdk-tools would inherit that migration transparently.

**Concern:** The framework has no mechanism to express cryptographic requirements. A contract cannot specify "this method requires a CRYSTALS-Dilithium signature" or "reject classical Ed25519 auth for high-value operations." The auth model is binary: authorized or not.

### The dependency chain

```
soroban-sdk-tools (#[auth])
  --> soroban_sdk::Address::require_auth()
    --> Soroban runtime auth framework
      --> Stellar consensus (SCP)
        --> Ed25519 (node identity)
        --> Ed25519/secp256k1/secp256r1 (account signatures)
```

A quantum break at any level of this chain would affect soroban-sdk-tools. But the critical distinction is:

- **Protocol-level break** (SCP node identity): catastrophic, framework irrelevant
- **Account-level break** (user signatures): mitigable through the framework's architecture

## 2. Quantum Threat Timeline Assessment

### Current consensus

NIST estimates that a CRQC capable of breaking Ed25519 (requiring ~2,300 logical qubits) could exist by 2030-2035. For secp256k1, the requirements are similar. The P-256 (secp256r1) curve, which Soroban recently added support for, is equally vulnerable.

### What this means for smart contracts

Unlike TLS sessions (which can be renegotiated) or file encryption (which can be re-encrypted), smart contract state is permanently on-chain. An ownership address recorded today as an Ed25519 public key will still be on-chain when quantum computers can forge signatures for that key.

The attack scenario:

1. Contract deployed in 2026 with `owner = Ed25519_PubKey_A`
2. CRQC operational in 2033
3. Attacker derives private key for `Ed25519_PubKey_A`
4. Attacker calls `transfer_ownership` with forged auth
5. `require_auth()` passes because the Soroban runtime validates the forged signature
6. Contract is compromised

### The window of vulnerability

The critical metric is not "when will quantum computers break crypto" but "how long will contracts deployed today be active." DeFi contracts often run for years. A mobile money contract (per Babatunde's use case) might run for decades. Infrastructure contracts may be effectively permanent.

## 3. Analysis of the Auth Architecture

### The Provider pattern as a migration vector

The Provider pattern is the most quantum-relevant feature in soroban-sdk-tools. Consider:

```rust
pub struct SingleOwner;
impl OwnableInternal for SingleOwner {
    fn owner(env: &Env) -> Address {
        OwnableStorage::get_owner(env).unwrap()
    }
}

// Future: quantum-resistant owner verification
pub struct PostQuantumOwner;
impl OwnableInternal for PostQuantumOwner {
    fn owner(env: &Env) -> Address {
        // Resolve owner through a quantum-resistant identity registry
        let pq_identity = PQIdentityRegistry::resolve(env, "owner");
        pq_identity.classical_address() // bridge to Soroban's Address type
    }
}
```

The swap from `SingleOwner` to `PostQuantumOwner` is a one-line change. This is excellent for contract-level migration. However, the fundamental problem remains: `require_auth()` still validates classical signatures at the runtime level.

### The `#[auth]` attribute's limitation

The `#[auth(Self::owner)]` pattern conflates two concerns:

1. **Identity resolution:** Who should be authorized? (answered by `Self::owner()`)
2. **Signature verification:** Is this really them? (answered by `require_auth()`)

In a post-quantum world, these must be separable. A contract might want to:

- Resolve identity through a quantum-resistant identity registry
- Verify a CRYSTALS-Dilithium signature instead of Ed25519
- Require both classical and post-quantum signatures during a transition period (hybrid mode)
- Accept different signature schemes based on the operation's value threshold

The current `#[auth]` attribute has no expressiveness for any of these requirements.

### Proposal: Extended auth attributes

```rust
#[contracttrait]
pub trait SecureOwnable {
    fn owner(env: &Env) -> Address;

    #[auth(Self::owner, scheme = "any")]           // current behavior
    fn read_config(env: &Env) -> Config;

    #[auth(Self::owner, scheme = "pq-required")]   // future: require PQ sig
    fn transfer_ownership(env: &Env, new_owner: Address);

    #[auth(Self::owner, scheme = "hybrid")]        // transition: both sigs required
    fn upgrade_contract(env: &Env, new_wasm: Bytes);
}
```

This would require changes to both soroban-sdk-tools and the Soroban runtime, but the macro architecture could generate the appropriate verification code today (even if `scheme` is currently a no-op that prepares for future runtime support).

## 4. The Sealed Auth Pattern Under Quantum Attack

### Current security model

The `impl_ownable!` macro generates:

```rust
#[contractimpl]
impl MyContract {
    pub fn transfer_ownership(env: Env, new_owner: Address) {
        let __auth_addr = <SingleOwner as OwnableInternal>::owner(&env);
        __auth_addr.require_auth();
        <SingleOwner as OwnableInternal>::transfer_ownership(&env, new_owner);
    }
}
```

This is non-overridable. The auth check is baked into the WASM. This is a strength against developer error and a vulnerability against quantum attack: there is no way to upgrade the auth mechanism without redeploying the contract.

### The upgradeability tension

Post-quantum migration requires one of:

1. **Contract redeployment:** Deploy a new contract with PQ auth, migrate state
2. **WASM upgrade:** Replace the contract's WASM while preserving state and address
3. **Runtime upgrade:** Stellar/Soroban upgrades `require_auth()` to be quantum-resistant
4. **Proxy pattern:** Auth delegated to an upgradeable verifier contract

Option 3 is the cleanest but depends on the Stellar Development Foundation. Option 2 is viable if the contract supports `upgrade()`. Options 1 and 4 require explicit planning.

### Recommendation

The sealed auth pattern should generate an `upgrade` method by default (or make it trivial to add one):

```rust
impl_ownable!(MyContract, SingleOwner, upgradeable = true);

// Generates:
pub fn upgrade(env: Env, new_wasm_hash: BytesN<32>) {
    let __auth_addr = <SingleOwner as OwnableInternal>::owner(&env);
    __auth_addr.require_auth();
    env.deployer().update_current_contract_wasm(new_wasm_hash);
}
```

Without upgradeability, sealed contracts become quantum-vulnerable tombs. The irony would be that the feature designed to prevent auth bypass (sealed auth) also prevents auth migration.

## 5. Key Rotation and Identity Continuity

### The problem

When a quantum computer can derive Ed25519 private keys from public keys, every address on Stellar is compromised. The question is not "if" but "how fast can keys be rotated?"

### Current support in soroban-sdk-tools

The `transfer_ownership` method provides key rotation. But:

- It requires auth from the current owner (which may already be compromised)
- There is no pre-commitment scheme (e.g., "rotate to this PQ key in 30 days unless I cancel")
- There is no dead man's switch (e.g., "if I don't re-authorize within 90 days, freeze the contract")
- There is no social recovery (e.g., "3 of 5 guardians can rotate ownership")

### What OZ does better here

OpenZeppelin's two-step transfer (transfer + accept) provides a window for error correction. If ownership transfer requires the new owner to accept, a quantum attacker who forges a `transfer_ownership` call still needs to control the new owner address's key to complete the transfer.

The OZ comparison acknowledges this: "Two-step transfers -- OZ's transfer_ownership + accept_ownership pattern prevents mistakes." But it understates the security benefit. Two-step transfer is not just about preventing mistakes -- it is a defense-in-depth mechanism that buys time during a cryptographic transition.

### Recommendation

Implement a `TwoStepOwner` provider as a first-class offering:

```rust
pub struct TwoStepOwner;
impl OwnableInternal for TwoStepOwner {
    fn owner(env: &Env) -> Address { /* current owner */ }
    fn transfer_ownership(env: &Env, new_owner: Address) {
        // Set pending, do not transfer yet
        PendingStorage::set_pending_owner(env, &new_owner);
    }
}

// Additional trait for acceptance
#[contracttrait]
pub trait OwnableAcceptable: Ownable {
    #[auth(Self::pending_owner)]
    fn accept_ownership(env: &Env);
    fn pending_owner(env: &Env) -> Address;
}
```

This is not just good practice -- it is a quantum migration defense layer.

## 6. The AuthClient and Quantum-Safe Testing

### Current testing model

The `AuthClient` supports:

```rust
auth_client.transfer_ownership(&new_owner)
    .authorize(&owner)
    .invoke();
```

And the blog mentions future support for:

```rust
auth_client.transfer_ownership(&new_owner)
    .sign(&keypair)      // Ed25519, secp256k1, P-256
    .invoke();
```

### Post-quantum testing

When PQ signature schemes are available, the `AuthClient` should support:

```rust
auth_client.transfer_ownership(&new_owner)
    .sign_pq(&dilithium_keypair)
    .invoke();

// Hybrid mode testing
auth_client.transfer_ownership(&new_owner)
    .sign(&ed25519_keypair)
    .sign_pq(&dilithium_keypair)
    .invoke();
```

The `CallBuilder` architecture (boxed closures wrapping invocations) is flexible enough to support this, but the interface should be designed now even if the underlying crypto is not yet available.

### Recommendation

Reserve the API surface for post-quantum auth methods in `CallBuilder`. Even stub implementations (that currently just delegate to classical auth) establish the migration path and signal to adopters that quantum readiness is on the roadmap.

## 7. Storage Key Vulnerability

### A subtle concern

The `SingleOwner` provider stores the owner as:

```rust
env.storage().instance().get(&Symbol::new(env, "owner"))
```

The storage key "owner" is a well-known string. If a quantum attacker can compromise the contract's identity (not just user keys), they could potentially manipulate storage directly through the Soroban runtime.

This is a protocol-level concern, not a framework-level one. But soroban-sdk-tools should document the threat model boundary: "This framework assumes the Soroban runtime's storage isolation is inviolable. If quantum attacks compromise runtime integrity (not just user signatures), all bets are off."

## 8. Cryptographic Agility Score

I evaluate frameworks on a "cryptographic agility" rubric:

| Criterion | Score | Notes |
|-----------|-------|-------|
| Signature scheme independence | 9/10 | Delegates entirely to `require_auth()` |
| Auth mechanism upgradeability | 4/10 | Sealed pattern prevents hot-swapping auth |
| Key rotation support | 5/10 | Basic transfer exists, no two-step or social recovery |
| Identity model extensibility | 7/10 | Provider pattern enables custom identity resolution |
| Transition period support | 2/10 | No hybrid auth, no gradual migration mechanisms |
| Testing for future schemes | 3/10 | AuthClient is extensible but has no PQ stubs |
| Documentation of threat model | 3/10 | No mention of quantum threats or crypto agility |

**Overall cryptographic agility: 4.7/10**

This is not alarming -- most smart contract frameworks score similarly. But it represents an opportunity. Because soroban-sdk-tools' architecture (Provider pattern, macro-generated auth, `CallBuilder`) is inherently more flexible than most, a small set of additions could raise this score to 7-8/10 and position the framework as a leader in quantum readiness.

## 9. Recommended Migration Roadmap

### Phase 1: Awareness (now)

- Document the quantum threat model in a security considerations section
- Acknowledge that `require_auth()` is only as secure as the underlying signature scheme
- Recommend two-step transfers as a defense-in-depth measure

### Phase 2: Architecture (6-12 months)

- Implement `TwoStepOwner` provider with timelock
- Add `upgrade()` support to sealed auth macros
- Reserve API surface in `AuthClient` for PQ methods
- Design `#[auth(Self::owner, scheme = X)]` attribute syntax

### Phase 3: Integration (when PQ signatures are available in Stellar)

- Implement PQ signature verification in `CallBuilder`
- Support hybrid auth (classical + PQ) during transition
- Provide migration tooling for existing contracts

### Phase 4: Enforcement (post-transition)

- Default to PQ-required auth for new contracts
- Emit warnings for contracts still using classical-only auth
- Provide automated migration scripts

## 10. What the Framework Gets Right

Despite my concerns, several architectural decisions position soroban-sdk-tools well for the quantum transition:

1. **No embedded cryptography.** The framework never touches keys, signatures, or hash functions. It delegates entirely to the Soroban runtime. This means it does not need to be patched when crypto changes -- only the runtime does.

2. **The Provider pattern is the right abstraction.** Swapping from `SingleOwner` to `PostQuantumOwner` is a type-level change, not a code rewrite. This is exactly the kind of modularity that smooth cryptographic transitions require.

3. **The two-trait architecture (Internal + Outer) separates concerns cleanly.** Business logic (Internal) never touches auth. Auth enforcement (Outer) never touches business logic. When the auth layer needs to change, business logic is unaffected.

4. **The macro-based approach means contracts can be regenerated.** When the `#[contracttrait]` macro is updated to support PQ auth attributes, all contracts that use it can regenerate their auth wrappers by simply recompiling. No manual code changes needed.

The foundation is sound. The urgency is in closing the gaps identified above before the quantum transition catches the ecosystem off-guard. History teaches us that cryptographic transitions take 10-15 years in practice (see: SHA-1 deprecation, TLS 1.0/1.1 sunset). Starting now is not premature -- it is precisely on schedule.
